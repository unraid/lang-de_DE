Ein Unraid Array besteht aus Einem oder Zwei Paritätsdatetnrägern und einer beliebigen Anzahl von Daten-Datenträgern. Die

**Färbige Status Anzeige** Bedeutung des Farbindikators am Anfang jeder Zeile von *Pool Datetnräger*:

**Pool Datenträger** ist ein einzelner oder ein Pool aus mehreren Datenträgern, *außerhalbe* des Unraid Arrays. Kann für
den Zugriff über eine Freigabe eingerichtet werden. Außerhalb des Arrays daher wesentlich höhere Schreibgeschwindigkeit.

Es gibt zwei möglichkeiten Pool Datenträger zu konfigurieren:

1. Als einzelnen Datenträger, oder
2. Als aus mehreren bestehenden Datenträgern.

Wenn als einzelner Datenträger konfiguriert kann der Datenträger mit jedem unterstützem Dateisystem formatiert (btrfs, reiserfs,
oder xfs) werden.  Diese Konfiguration ist die Performanteste jedoch auch die Riskanteste - wenn der Datenträger
ausfällt sind möglicherweise alle Daten verloren.

Wenn als aus mehreren bestehenden Datenträger konfiguriert, wird Unraid automatisch *btrfs-raid1* wählen (jeweils für Daten
und Meta-Daten).  btrfs erlaubt es eine beliebige Anzahl von Datenträgern hinzuzufügen und garantiert das jede Datei auf zwei
verschiedenen Datenträger geschrieben wird. Der Pool kann somit einem einzelnen Datenträgerausfall wiederstehen.

Wenn [User Shares](/Settings/ShareSettings) aktiviert sind, werden Benutzer-Freigaben am Array autmoatisch mit
dem Cache-Pool verknüpft um die Schrebigeschwindigkeit zu erhöhen.
Ein Spezieller Prozess namens *mover* kann für die Ausführung, in regelmäßigen
Abständen, für die Übertragung vom Cache-Pool zum Array geplant werden.
:end

:array_devices_help:
#### Datenträger zuweisen

Ein Unraid Array besteht aus Einem oder Zwei Paritätsdatetnrägern und einer beliebigen Anzahl von Daten-Datenträgern. Die
Daten-Datenträger sind nur für Nutzerdaten und der/die Paritätsdatentäger dient rein der Redundanz die zur
Datenwiederherstellung nötig ist falls ein Daten-Datenträger ausfällt.

Beachten Sie, dass wen wir den Begriff *Datenträger* verwenden wenn wir uns auf einen Arrag Datenträger beziehen. Wir
verwenden auch manchmal *Festplatte* (oder unter Umständen *Laufwerk*) wenn wir uns auf einen Datenträger (HDD) beziehen.
Dies liegt daran das in RAID Systemen ein Lesen/Schreiben auch bei einem fehlendem bzw. Deaktivieren Datenträger möglich
ist! Darüber hinaus kann es Hilfreich sein zu Fragen "Welcher Datenträger wird als Parität verwendet?" oder "Welcher
Datenträger entspricht disk2?".

Wir brauchen daher eine möglichkeit Datenträger dem Array zuzuteilen. Dies kann auf der Hauptseite wenn das Arry
gestoppt ist durchgeführt werden. Für jeden Array-Datenträger gibt es ein Drop-Down Feld in dem alle nicht zugewiesenen
Datenträger angezeigt werden. Um einen Datenträger zuzuweisen, wählen Sie einfach einen Datenträger aus der Liste aus.
Jedes mal wenn eine Zuweisung durchgeführt wurde aktualisiert das System eine Konfigurationsdatei um die Änderung zu speichern.

#### Anforderungen

Ander als traditionelle RAID Systeme die Daten über mehrer Datenträger aufsplitten speicher der Unraid Server
die Daten individuell auf den Datenträgern. Folglich müssen bei allen Schrebivorgängen der Datenträger auf den
eine Datei geschrieben wird und der Paritätsdatenträger aktiv sein. Aus diesen Gründen

* muss der Paritätsdatenträger der größte oder größer als der größte Daten-Datenträger sein

und

* sollte wenn möglich der schnellste/performanteste Datenträger sein.

#### Allgemeine Richtlinien

Diese Schritte sollen Ihnen bei der Erstellung eines Unraid Arrays helfen:

1. Welcher Datenträger soll für die Parität benutzt werden und welche Datenträger sollen als Daten-Datenträger benutzt
werden. Mit disk1, disk2, usw. beschriften. Notieren sie die Seriennummern der Datenträger,
Sie werden sie später brauchen.

2. Bauen Sie die Datenträger in den Server ein und Starten Sie Unraid und navigieren Sie zur WebGUI. Sollte dies eine neue
Installation sein wird die Hauptseite keine installierten Datenträger anzeigen. Dies heißt aber nicht das Ihre Datenträger
nicht erkannt wurden, Sie wurden lediglich noch nicht zugewiesen.

3. Nun brauchen Sie die Seriennummern aus Schritt 1. Wählen Sie für die Paritäts- und Daten-Datenträger
den Datentäger, basierend auf der Serienummer, aus dem Drop-Down aus und weisen ihm den dementsprechenden Slot zu.

#### Hot Plug

Sie können die Datenträger ebenseo mittels *Hot Plug* einsetzen wenn Ihre Hardware dies unterstützt. Wenn Sie zum
Beispiel Festplatten Wechselrahmen benutzen können Sie diese im Laufenden Betrieb einsetzen wenn das Array gestoppt
ist. Warten Sie einen kurzen Moment und Aktualisieren sie die Hauptseite von Unraid und der soeben eingesetzte Datenträger
erscheint im Drop-Down.

#### Weiterführende Schritte

Wenn Sie alle Datenträger zugewiesen haben bitte folgen sie dem Array Status Abschnitt
und starten Sie das Array.
:end

:encryption_help:
#### Verschlüsselung

Wenn das Array gestoppt ist kann ein neuer Verschlüsselungsschlüssel vergeben werden. Bitte beachten Sie wenn ein Datetnträger
bereits mit einem anderen Schlüssel verschlüsselt wurde das er mit einem neuen nicht starten wird. Die Änderung des Schlüssels
von bereits verschlüsselten Datenträgern erfordert eine neuerliche formatierung **dies löscht alle bereits vorhanden Daten!**

#### Passphrase

Eine Passphrase besteht aus bis zu 512 Zeichen. Es wird empfohlen nur die 95 druckbaren Zeichen der 128 ersten Zeichen
der [ASCII table](https://en.wikipedia.org/wiki/ASCII) zu benutzen, da diese immer die gleiche Binäre Bedeutung haben.
Andere Zeichen können je nach System anders Codiert sein und könnten somit die gültigkeit der Passphrase beeinträchtigen.
Falls Sie eine längere Passphrase möchen oder Binäre Dateien inkludieren wollen, laden sie bitte ein Schlüssel-Datei hoch.

Bitte [cryptsetup FAQ](https://gitlab.com/cryptsetup/cryptsetup/wikis/FrequentlyAskedQuestions#5-security-aspects)
für die Definition einer *sicheren* Passphrase beachten.

**Prägen** Sie sich die Passphrase ein. **FALLS SIE VERLOREN WIRD, KANN DER VERSCHLÜSSELTE INHALT NICHT WIEDERHERGESTELLT WERDEN!**

#### Schlüssel-Datei

Wählen Sie eine Schlüsseldatei mit einem gespeicherten Schlüssel oder einer Binärdatei aus. Die maximale Größe beträgt 8M (8388608 byte).

**Sichern** Sie unbedingt Ihre Schlüsseldatei. **FALLS SIE VERLOREN WIRD, KANN DER VERSCHLÜSSELTE INHALT NICHT WIEDERHERGESTELLT WERDEN!**
:end

:info_btrfs_balance_help:
**Balance** löst ein *btrfs balance* aus das die Daten über alle Pool Datenträger Restriped, zum Beispiel,
um den Pool von raid1 zu raid0 zu konvertieren oder umgekehrt.

Wenn ein *full balance* ausgeführt wird, wird Grundsätzlich alles im Dateisystem neu geschrieben.

Eine *mode conversion* wirkt sich auf die Datenbereiche von btrfs aus; Meta-Daten nutzen immer raid1 und werden falls nötig zu raid1 konvertiert im Zuge der Balance.

Dies kann eine lange Zeit dauern, abhängig von der Größe des Dateisystems und der Geschwindigkeit des Datenträgers.

Unraid verwendet die Standardoptionen beim Erstellen eines Pools mit mehreren Geräten:

`-dconvert=raid1 -mconvert=raid1`

Eine ausführlichere Dokumentation zu btrfs-balance finden Sie unter [Manpage](https://btrfs.wiki.kernel.org/index.php/Manpage/btrfs-balance)

*Hinweis: raid5 und raid6 sind von der Linux Community immer noch als **experimentell** eingestuft*
:end

:share_edit_allocation_method_help:
Diese Einstellung bestimmt wie und auf welchen Datenträgern Unraid neue Dateien/Verzeichnisse erstellt:

**High-Water**
Benutzt den ersten Datenträger der noch genug Speicherplatz unter der *High-Water Grenze* besitzt. Die
*High-Water Grenze* wird durch den Größten Daten-Datenträger geteilt durch zwei festgelegt. Wenn kein Datenträger
Speicherplatz über der *High-Water Grenze* zur Verfügung hat wird die *High-Water Grenze* durch zwei geteilt und erneut geprüft.

Das Ziel von **High-Water** ist möglichst viele Daten auf jeden Datenträger zu schreiben (um die Anlaufvorgänge der
Datenträger zu minimieren) und gleichzeitig die Datenträgerauslastung von allen Datenträgern auf gleichem
niveaus zu halten (um Daten gleichmäßig auf dem Array zu verteilen).

**Fill-Up**
Benutzt den ersten Datenträger der noch freien Speicherplatz über der **Minimale freier Speicherplatz**
Einstellung hat.

**Most-Free**
Benutzt den Datenträger der am meisten freien Speicherplatz hat.
:end

:share_edit_split_level_help:
Gibt an ob ein Verzeichnis über meherer Datenträger aufgeteilt werden darf.

**Jedes Verzeichnis automatisch Splitten falls erforderlich**
Wenn eine neue Datei oder ein neues Unterverzeichnis in einer Freigabe erstellt wird, wählt Unraid OS zuerst den Datenträger
anhand der Konfigurierten *Zuweisungsmethode* aus.  Wenn das übergeordnete Verzeichnis mit der neu zu erstellenden Datei
oder Verzeichnis nicht auf dem Datenträger existiert, erstellt Unraid zuerst alle erforderlichen übergeordneten Verzeichnisse
und anschließend die neue Datei bzw. das neue Unterverzeichnis.

**Hauptverzeichnis automatisch Splitten falls erforderlich**
Wenn eine neue Datei oder ein Unterverzeichnis im Hauptverzeichnis der Freigabe erstellt wird und das Unterverzeichnis
auf dem Datenträger des Hauptverzeichnises nicht vorhanden ist, wird zuerst das Unterverzeichnis erstellt.
Wenn eine neue Datei oder ein Unterverzeichnis im zweiten oder weiteren Unterverzeichnisen einer Freigabe erstellt wird, wird
die neu zu erstellende Datei oder Verzeichnis auf dem selben Datenträger wie das Hauptverzeichnis erstellt.

**Verzeichnis nach "N" Ebenen Splitten falls erforderlich**
Ähnlich wie oberhalb: Wenn eine neue Datei oder ein Unterverzeichnis erstellt wird und das übergeordnete Verzeichnis auf Ebene
"N" befindet und auf dem ausgewählten Datenträger nicht vorahnden ist, erstellt Unraid alle übergeordneten Verzeichnisse. Wenn
das übergeordnete Verzeichnis der neuen Datei oder Verzeichnis sich nach der Ebene "N" befindet, dann wird die neue Datei oder
Verzeichnis auf dem selben Datenträger erstellt auf dem sich das übergeordnete Verzeichnis befindet.

**Manuell: Verzeichnisse nicht automatisch Splitten**
Wenn eine neue Datei oder ein Unterverzeichnis auf deinem Datenträger erstellt werden muss erstellt Unraid das Verzeichnis
nur auf dem Datenträger auf denen das Hauptverzeichnis vorhanden ist.
:end

:share_edit_cache_pool_help:
Gibt an ob neue Dateien und Verzeichnisse auf den Cache-Datenträger/Pool geschrieben werden dürfen falls dieser vorhanden ist.
Diese Einstellung beeinflusst das verhalten des *Mover*.

**Nein** verbietet das Schreiben neue Dateien und Verzeichnisse auf den Cache-Datenträger/Pool.
Keine Aktivität vom *Mover*, ignoriert alle Dateien die sich eventuell noch auf dem Cache-Datenträger/Pool befinden.

**Ja** gibt an das neue Dateien und Verzeichnisse auf den Cache-Datenträger/Pool geschrieben werden sollen, vorausgesetzt
es befindet sich genug freier Speicherplatz auf dem Cache-Datenträger/Pool.
Wenn nicht genug Speicherplatz auf dem Cache-Datenträger/Pool vorhanden ist werden neue Dateien und Verzeichnisse direkt auf
das Array geschrieben. Wenn der *Mover* aktiv ist werden die Dateien vom Cache-Datenträger/Pool auf das Array transferiert.

**Nur** gibt an das neue Dateien und Verzeichnisse auf den Cache-Datenträger/Pool geschrieben werden müssen.
Wenn nicht genug Speicherplatz auf dem Cache-Datenträger/Pool vorhanden ist werden, schlagen *Schreibvorgänge* mit dem Fehler
*kein freier Speicherplatz vorhanden* fehl. Keine Aktivität vom *Mover* ignoriert alle Dateien die sich eventuell noch auf dem Array befinden.

**Bevorzugt** gibt an das neue Dateien und Verzeichnisse auf den Cache-Datenträger/Pool geschrieben werden sollen,
vorausgesetzt es befindet sich genug freier Speicherplatz auf dem Cache-Datenträger/Pool.
Wenn nicht genug Speicherplatz auf dem Cache-Datenträger/Pool vorhanden ist werden neue Dateien und Verzeichnisse direkt auf
das Array geschrieben. Wenn der *Mover* aktiv ist werden die Dateien vom Array auf den Cache-Datenträger/Pool transferiert.

**HINWEIS:** Der Mover kann nur Dateien verschieben die gerade nicht geöffnet/bentutzt werden.
Das heißt wenn Sie Dateien von System Anwendungen wie zB Docker oder VMs verschieben wollen müssen diese Anwendungen
vorher gestoppt werden wenn der Mover läuft.
:end

:smb_case_sensitive_names_help:
Gibt an ob bei Dateinamen zwischen Groß- und Kleinschreibung unterschieden wird.

Die Standardeinstellung **Auto** erlaubt Clienten die zwischen Groß- und Kleinschreibung unterschieden können (Linux CIFSVFS)
dem Samba Server paketweise mitzuteilen,dass sie zwischen Groß- und Kleinschreibung auf das Dateisystem zugreifen möchten (Unterstützung für UNIX
Groß- und Kleinschreibung). Kein Windows-System unterstützt Dateinamen, die zwischen Groß- und Kleinschreibung unterschieden können daher ist das Festlegen der Option auf **Auto** gleich
wie Nein, die Dateinamen die von Windows stammen werden aber richtig an den Server übergeben.  Diese Einstellung kann die
Leistung bei großen Verzeichnissen beeinträchtigen da Samba eine Dateisuche durchführt und mit übergebenen Namen übereinstimmen muss.

Die Einstellung **Ja** gibt an das die Dateienamen vom Clienten übernommen werden und auch so gespeichert werden und nur so abrufbar sind.
Dies beschleunigt den Zugriff auf sehr große Verzeichnisse einige Windows-Anwendungen funktionieren mit dieser Einstellung möglicherweise nicht ordnungsgemäß.
Wenn zum Beispiel "MeineDatei" erstellt wird, eine Windows Anwendung jedoch versucht "MEINEDATEI" zu öffnen (in Windows zulässig), wird diese nicht gefunden.

Die Einstellung **Nur Kleinschreibung**gibt an die Dateinamen, nicht nur neuer Datein, werden in Kleinschreibung umgewandelt
Mit anderen Worten werden alle Dateinamen die von Windows erstellt werden in Kleinbuchstaben gespeichert und abgerufen. Dies
stellt sicher das alle Windows Anwendungen, unabhängig der Groß- und Kleinschreibung alle Dateien ordnungsgemäß finden, diese Einstellung beinhaltet nicht den jedoch die Ordnerliste.
Beachte das diese Einstellung nur für neue Freigabe gesetzt werden sollte.
:end

:disk_default_partition_format_help:
Definiert die Art des Partitionslayouts, das beim Formatieren von Festplatten mit einer Größe von **nur** 2 TB und kleiner
erstellt werden soll. (Alle Geräte größer als 2TB werden immer mit GPT-Partitionstabellen formatiert)

**MBR: unaligned** Diese Einstellung erstellt eine Partitionstabelle im MBR-Stil, wobei
die einzelne Partition 1 im **63. Sektor** vom Anfang der Platte an beginnt.  Dies ist die *traditionelle*
Einstellung für praktisch alle Partitionstabellen im MBR-Stil.

**MBR: 4K-aligned** Diese Einstellung erstellt eine Partitionstabelle im MBR-Stil, wobei 
die einzelne Partition 1 im **64. Sektor** vom Anfang der Platte an beginnt. Da die Sektorgröße 512 Bytes beträgt,
wird der Anfang von Partition 1 auf einer 4K-Byte-Grenze *eingerichtet*.  Dies ist erforderlich für eine korrekte
Unterstützung von sogenannten *Advanced Format*-Laufwerken.

Ändern Sie diese Einstellung nicht von der Standardeinstellung, es sei denn, Sie haben eine spezielle Anforderung jenseits des **MBR: 4K-aligned** Standards.
:end

:mgmt_certificate_expiration_help:
**Provision** may be used to allocate a *free* SSL Certficiate from [Let's Encrypt](https://letsencrypt.org/) and
then upload to your server. Note: We **highly** recommend using a static IP address in this case.

**Update DNS** may be used to manually initiate updating the DNS A-record of your server FQDN on unraid.net. Note
that DNS propagation change could take anywhere from 1 minute to several hours (we set TTL to 60 seconds).

Note: **Provision** may fail if your router or upstream DNS server has
[DNS rebinding protection](https://en.wikipedia.org/wiki/DNS_rebinding) enabled. DNS rebinding
protection prevents DNS from resolving a private IP network range. DNS rebinding protection is meant as
a security feature on a local LAN which includes legacy devices with buggy/insecure "web" interfaces.

One source of DNS rebinding protection could be your ISP DNS server. In this case the problem may be solved by
switching to a different DNS server such as Google's public DNS.

More commonly, DNS rebinding protection could be enabled in your router. Most consumer routers do not implement DNS
rebinding protection; but, if they do, a configuration setting should be available to turn it off.

Higher end routers usually do enable DNS rebinding protection however. Typically there are ways of turning it off
entirely or selectively based on domain. Examples:

**DD-WRT:** If you are using "dnsmasq" with DNS rebinding protection enabled, you can add this line to your router
configuration file:

`rebind-domain-ok=/unraid.net/`

**pfSense:** If you are using pfSense internal DNS resolver service, you can add these Custom Option lines:

`server:`<br>
`private-domain: "unraid.net"`

**Ubiquiti USG router:** you can add this configuration line:

`set service dns forwarding options rebind-domain-ok=/unraid.net/`

**OpenDNS:** Go to Settings -> Security and *remove* the checkbox next to
 "Suspicious Responses - Block internal IP addresses". It is an all-or-nothing setting.

When all else fails, you can create an entry in your PC's *hosts* file to override external DNS and
directly resolve your servers unraid.net FQDN to its local IP address.
:end

:nfs_tunable_fuse_remember_help:
When NFS is enabled, this Tunable may be used to alleviate or solve instances of "NFS Stale File Handles"
you might encounter with your NFS client.

In essence, (fuse_remember) tells an internal subsystem (named "fuse") how long to "remember" or "cache"
file and directory information associated with user shares.  When an NFS client attempts to access a file
(or directory) on the server, and that file (or directory) name is not cached, then you could encounter
"stale file handle".

The numeric value of this tunable is the number of seconds to cache file/directory name entries,
where the default value of 330 indicates 5 1/2 minutes.  There are two special values you may also set
this to:

* 0 which means, do not cache file/directory names at all, and
* -1 which means cache file/directory names forever (or until array is stopped)

A value of 0 would be appropriate if you are enabling NFS but only plan to use it for disk shares,
not user shares.

A value of -1 would be appropriate if no other timeout seems to solve the "stale file handle" on
your client.  Be aware that setting a value of -1 will cause the memory footprint to grow by approximately
108 bytes per file/directory name cached.  Depending how much RAM is installed in your server and how many
files/directories you access via NFS this may or may not lead to out-of-memory conditions.
:end

:eth_enable_bonding_help:
Bonding ist eine Funktion, die mehrere Ethernet-Schnittstellen zu einer einzigen *Bonded* Schnittstelle mit dem Namen **bond0** zusammenfügt.
Dies kann verwendet werden, um die Verbindungsredundanz und/oder den Durchsatz des Systems zu erhöhen.
Es werden verschiedene Verbindungsmodi unterstützt (siehe unten), einige Modi erfordern jedoch die Unterstützung vom Netzwerkswitch.
:end

:eth_bonding_mode_help:
**Modus 0 (balance-rr)**<br>
Dieser Modus überträgt Pakete in einer sequentiellen Reihenfolge vom ersten verfügbaren Slave bis zum letzten.
Wenn zwei Schnittstellen im Bond zu den Slaves gehören und zwei Pakete aus der Bonded Schnittstelle kommen, wird das erste auf dem ersten Slave und der zweite Frame auf dem zweiten Slave übertragen. Das dritte Paket wird am ersten gesendet und so weiter.
Dies bietet Lastausgleich und Fehlertoleranz.

**Modus 1 (active-backup) - standard**<br>
Dieser Modus versetzt eine der Schnittstellen in einen Ausfallsicherungsstatus und aktiviert sie nur, wenn die Verbindung durch die aktive Schnittstelle verloren geht.
Es ist jeweils nur ein Slave im Bond aktiv. Ein anderer Slave wird erst aktiv, wenn der aktive Slave ausfällt.
Dieser Modus bietet Fehlertoleranz.

**Modus 2 (balance-xor)**<br>
Dieser Modus überträgt Pakete basierend einer XOR-Formel. Die Quell-MAC-Adresse wird mit der Slave-Anzahl des Ziel-MAC-Adressmoduls XOR-verknüpft.
Dies wählt für jede Ziel-MAC-Adresse denselben Slave aus und bietet Lastausgleich und Fehlertoleranz.

**Modus 3 (broadcast)**<br>
Dieser Modus überträgt alles auf allen Slave-Schnittstellen. Dieser Modus wird am wenigsten verwendet (nur für bestimmte Zwecke) und bietet nur Fehlertoleranz.

**Modus 4 (802.3ad)**<br>
Dieser Modus wird als *Dynamic Link Aggregation* bezeichnet. Es werden Aggregationsgruppen erstellt, die dieselben Geschwindigkeits- und Duplexeinstellungen verwenden.
Ein Switch der die dynamische IEEE 802.3ad-Verbindung unterstützt ist erforderlich.
Die Slave-Auswahl für ausgehenden Datenverkehr erfolgt gemäß der Übertragungs-Hash-Richtlinie, die über die Option xmit_hash_policy von der Standard Einfachen-XOR-Richtlinie geändert werden kann.
Beachten Sie dass möglicherweise nicht alle Übertragungsrichtlinien 802.3ad-kompatibel sind insbesondere im Hinblick auf die Anforderungen an die Paktanforderungen in Abschnitt 43.2.4 des 802.3ad-Standards.
Unterschiedliche Implementierungen haben unterschiedliche Toleranzen für Nichteinhaltung.

**Modus 5 (balance-tlb)**<br>
Dieser Modus wird als *Adaptiver Sendelastausgleich* bezeichnet. Der ausgehende Verkehr wird entsprechend der aktuellen Last und Warteschlange auf jeder Slave-Schnittstelle verteilt.
Eingehender Verkehr wird vom aktuellen Slave empfangen.

**Modus 6 (balance-alb)**<br>
Dieser Modus wird als *Adaptiver Lastausgleich* bezeichnet. Dies beinhaltet balance-tlb + Receive Load Balancing (rlb) für IPV4-Verkehr.
Der Empfangslastausgleich wird durch ARP-Aushandlung erreicht.
Der Bindungstreiber fängt die vom Server auf dem Weg nach draußen gesendeten ARP-Antworten ab und überschreibt die src-hw-Adresse mit der eindeutigen hw-Adresse eines der Slaves im Bond
sodass verschiedene Clients unterschiedliche hw-Adressen für den Server verwenden.

*Modus 1 (active-backup) ist die empfohlene Einstellung. Andere Modi erlauben es Ihnen die Einstellungen an bestimmte Umgebungen anzupassen, erfordern jedoch dementsprechende Netzwerk Switches.
Die Auswahl eines nicht unterstützten Modus kann zu einer Unterbrechung der Netzwerkommunikation führen.*
:end

:sysdevs_iommu_groups_help:
Listet alle IOMMU Gruppen die im System verfügbar sind auf, mittels `lspci` werden zugehörige Informatinen zu den IOMMU Gruppen angezeigt.  Die Identifikationszahlen werden zur Konfiguration von PCI Passthrough verwendet.

Geräte die hier ausgewählt werden werden dem VIFO-PCI Treiber beim Booten zugeordnet, dies ermöglicht eine Zuordnung zu einer VM, und verbietet dem Kernel den Treiber automatisch auf dem Host System zu laden.

**Hinweis: Die Auswahl eines Geräts ordnet eventuell nicht nur dieses Gerät zu, sondern gilt für *alle* anderen Geräte in der IOMMU Gruppe in der sich das Gerät befindet.**

&nbsp;<i class="fa fa-circle orb green-orb middle"></i>&nbsp;Dieses Symbol zeigt an das das Gerät aktuell dem VIFO-PCI Treiber zugeordner ist.

&nbsp;<i class="fa fa-retweet grey-orb middle"></i>&nbsp;&nbsp;&nbsp;Dieses Symbol zeigt an das FLR (Function Level Reset) unterstützt wird.

<input type="checkbox" value="" disabled>&nbsp;&nbsp;Falls ein Auswahlfeld ausgegraut ist wir dieses Gerät von Unraid verwendet und kann somit nicht zur VM durchgereicht werden.
:end
